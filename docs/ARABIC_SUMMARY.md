# الملخص العربي للمشروع

## 1. فكرة المشروع

المشروع هو أداة لبناء حزمة حماية منزلية اعتمادًا على تصميم Figma المرفق. يختار المستخدم الكاميرات والـVariants والكميات، ثم يرى نفس الاختيارات فورًا في لوحة المراجعة مع تحديث الإجمالي والتوفير. البيانات الأصلية تأتي من Backend وليست مكتوبة داخل مكونات الواجهة.

## 2. هيكل الـFrontend والـBackend

المستودع Monorepo يحتوي على مساحتين:

- `client/`: تطبيق React وTypeScript مبني بواسطة Vite. المكونات قابلة لإعادة الاستخدام، التنسيق باستخدام CSS Modules وCSS custom properties، والحالة موجودة في Context واحد.
- `server/`: خادم Express وTypeScript. يقدم بيانات الحزمة من ملف JSON بعد التحقق منها باستخدام Zod، مع Helmet وCORS.

المجلد `docs/` يحتوي على مراجعة التصميم والأصول وتقارير المقارنة البصرية، بينما `scripts/` يحتوي على أداة إنشاء صورة الفرق البصري.

## 3. لماذا استخدمت useReducer

الحالة تحتوي على عمليات مترابطة: فتح خطوة، تغيير Variant نشط، زيادة أو تقليل كمية، وحالة الحفظ. `useReducer` يجمع هذه التحولات في مكان واحد، ويجعلها حتمية وقابلة للاختبار، ويمنع وجود منطق متفرق أو حالات كمية مكررة بين أجزاء الواجهة.

## 4. كيف تم تخزين كل Variant بكمية مستقلة

كل Variant له مفتاح كمية مستقل بالشكل:

```text
productId:variantId
```

مثال: الأبيض والأسود للمنتج نفسه يملكان مفتاحين منفصلين. المنتجات التي لا تملك Variants تستخدم:

```text
productId:default
```

اختيار لون مختلف يغيّر الـVariant النشط المعروض في الكارت فقط، ولا يمسح كمية أي لون آخر.

## 5. كيف تتم مزامنة Builder وReview Panel

لا توجد حالتان تحتاجان إلى مزامنة. Stepper الموجود في الكارت وStepper الموجود في لوحة المراجعة يرسلان نفس الإجراء إلى نفس Reducer باستخدام نفس مفتاح الكمية. كلا الجانبين يقرآن القيمة الجديدة مباشرة من `BundleState`، لذلك لا يوجد `useEffect` للنسخ أو المزامنة ولا احتمال لاختلاف القيمتين.

## 6. كيف يتم حساب selected count

العدد يحسب المنتجات المميزة التي أصبح مجموع كمياتها أكبر من صفر. المنتج الذي له أكثر من Variant يُحسب مرة واحدة مهما كان عدد الـVariants المختارة أو عدد الوحدات. لذلك القيمة تعني “عدد المنتجات المختلفة المختارة”، وليست عدد الوحدات.

## 7. كيف يتم حساب total وsavings

الـSelectors تنشئ صفوف المراجعة من الحالة الحالية، ثم تضرب سعر الوحدة الحالي وسعر المقارنة في الكمية. بعد ذلك:

```text
total = مجموع الأسعار الحالية للصفوف المحتسبة
savings = مجموع أسعار المقارنة - total
```

الشحن ظاهر في المراجعة لكنه غير مضاف إلى الإجمالي بحسب بيانات Figma الأصلية.

## 8. كيف يعمل localStorage

زر الحفظ يكتب نسخة مرقمة `version: 1` تحت المفتاح `bundle-builder:configuration`. عند إعادة التحميل يتم التحقق من رقم النسخة، ومعرّفات الخطوات، وكل مفاتيح الكميات، وأن الكميات أعداد صحيحة غير سالبة، وأن المنتجات الإجبارية لا تقل عن الحد الأدنى، وأن كل Variant نشط موجود فعلًا. أي بيانات ناقصة أو تالفة أو قديمة يتم تجاهلها والعودة إلى حالة Figma الابتدائية.

## 9. كيف تمت مطابقة Figma

تم استخدام القياسات والأصول والنصوص والأسعار المستخرجة من عقدتي Desktop وMobile. التخطيط يستخدم Grid وFlexbox مع القيم المقاسة في Tokens وCSS Modules، وليس صفحة مطلقة المواضع مولدة من Figma. تم تنفيذ ثلاث جولات Desktop وجولة Responsive مع Screenshots وصور فرق حقيقية دون Tolerance.

تعذر الحصول على Screenshot كامل الدقة في جولة التنسيق النهائية بسبب نفاد حصة استدعاءات Figma MCP. لذلك صور المرجع الحالية مأخوذة من Thumbnail أصلي منخفض الدقة لنفس ملف Figma. نسب اختلاف البكسلات المسجلة لا تمثل دقة التصميم الفعلية لأن التكبير والضغط يؤثران عليها بقوة.

## 10. الاختبارات المكتوبة

- اختبارات Reducer لاستقلال كميات الـVariants.
- اختبارات تكامل لتزامن Steppers بين Builder وReview.
- اختبارات عدد المنتجات المختارة والإجمالي والتوفير.
- اختبارات حفظ واستعادة ورفض بيانات `localStorage` غير الصالحة.
- اختبارات فتح وإغلاق Accordion والانتقال للخطوة التالية.
- اختبارات Checkout Dialog.
- اختبارات API باستخدام Supertest.
- اختبارات Playwright للهندسة المقاسة، لوحة المفاتيح، Focus trap، Escape وإعادة التركيز، Accessibility، Reduced Motion، عدم وجود Horizontal Overflow، تحميل الصور، وعدم وجود أخطاء Console أو Page Errors.

## 11. طريقة تشغيل المشروع

من جذر المشروع:

```bash
npm install
npm run dev
```

يفتح الـFrontend افتراضيًا على `http://localhost:5173` والـBackend على `http://localhost:3001`. ويمكن تشغيلهما منفصلين:

```bash
npm run dev --workspace client
npm run dev --workspace server
```

تشغيل فحوص الجودة:

```bash
npm run lint
npm run test
npm run build
npm run test:e2e
```

## 12. القيود المتبقية

- ملفات خطوط Gilroy وTT Norms Pro التجارية لم تكن متاحة بترخيص ضمن Figma أو المشروع، لذلك استُخدمت بدائل نظام موثقة ولا يمكن ضمان تطابق الحروف 100%.
- يلزم إعادة المقارنة بصورة Figma كاملة الدقة عندما تصبح حصة MCP متاحة؛ لا توجد مطالبة بتطابق Pixel-perfect كامل حاليًا.
- Checkout يعرض Dialog تأكيد فقط، ولا ينشئ طلبًا أو عملية دفع لأن ذلك خارج النطاق.
- الحفظ محلي داخل المتصفح ولا يرتبط بحساب مستخدم أو قاعدة بيانات.
